## Purpose

Use this skill when a completed cycle produced durable technical knowledge worth saving for future sessions.

## Allowed content

- Architecture or design decisions
- Team conventions
- Reusable implementation patterns
- Technical gotchas
- Bugfix summaries and root causes

## Workflow

1. Start from the completed implementation or review output.
2. Extract only durable technical knowledge.
3. Remove business details, user details, and raw operational data.
4. Write the entry in English.
5. Keep it concise and useful for a future developer.

## Good entry shape

- Title: short and searchable
- What changed
- Why it matters
- Where it applies
- Any non-obvious lesson

## Hard rules

- Do not propose requirements text, code diffs, or raw transcripts.
- Do not include PII, policy numbers, claim identifiers, amounts, or customer identifiers.
- If unsure whether something is safe, do not propose it.
